# Startup UI/UX Design

Create interface work that feels like a real product made by a strong designer and frontend engineer, not a generic AI mockup.

This skill is inspired in part by the earlier interface-design skill’s strongest ideas: gather requirements first, generate multiple genuinely different options, compare them clearly, and synthesize the best final direction instead of locking into the first idea.

## Primary Goal

Produce UI/UX that is:

- visually polished
- structurally clear
- realistic for a startup product
- consistent across pages and states
- responsive and accessible
- implementable without breaking existing functionality

## Success Standard

A successful result should feel:

- intentional rather than autogenerated
- modern but not overdesigned
- clean, aligned, and trustworthy
- focused on the product’s main job
- coherent across layout, spacing, typography, color, states, and interactions

## Core Principles

### 1. Do not settle for the first idea
Generate at least 3 meaningfully different design directions before locking the final solution.

If sub-agents or parallel exploration tools are available, use them. If not, still reason through multiple alternatives manually.

Each direction must differ in a real way, not just by color or small layout changes. Differences may include:

- information hierarchy
- navigation model
- page density
- card vs table emphasis
- dashboard structure
- interaction pattern
- visual tone
- onboarding flow structure
- focus on analysis vs action

### 2. Preserve working product logic
When redesigning an existing interface:

- do not break working features just to restyle the UI
- keep important flows intact unless there is a strong UX reason to change them
- identify what is visual debt vs functional debt
- fix structure and usability before adding decoration

### 3. Prioritize clarity over decoration
Use visual styling to support understanding, not to distract.

Prefer:

- strong hierarchy
- good spacing
- readable typography
- clear primary actions
- obvious grouping
- consistent component behavior

Avoid:

- random gradients
- excessive glassmorphism
- too many accents competing for attention
- oversized shadows
- decorative elements with no UX purpose
- cluttered dashboards with equal emphasis everywhere

### 4. Make it feel like a real shipped product
Use realistic structure, labels, and content.

Do not produce pages full of vague placeholders such as:

- "Lorem ipsum"
- "Sample analytics"
- "Card title"
- "Insert data here"

Instead, use believable startup-oriented content and component labels that fit the product.

### 5. Design the states, not only the ideal screen
Every important flow should account for real UI states:

- default
- hover
- active
- focus
- selected
- disabled
- loading / skeleton
- empty
- success
- warning
- error
- no-results

A polished product is judged heavily by how these states are handled.

### 6. Responsive by design, not as an afterthought
Consider layout behavior from the start for:

- desktop
- tablet
- mobile

Specify what stacks, collapses, hides, scrolls, becomes tabbed, or moves into sheets/drawers.

### 7. Accessibility is part of good design
Ensure the design remains usable with:

- good color contrast
- visible focus states
- meaningful labels
- readable text sizes
- sensible hit targets
- keyboard-friendly interactions where relevant

### 8. Consistency beats novelty
A smaller number of well-used patterns is better than many one-off components.

Keep consistent:

- spacing scale
- border radius
- shadow depth
- input style
- button hierarchy
- icon sizing
- heading rhythm
- card behavior
- page paddings and gutters

## Default Design Taste

Unless the user explicitly asks otherwise, bias toward:

- one strong default theme instead of splitting effort across light and dark modes
- a clean modern SaaS aesthetic
- restrained but premium visuals
- medium density layouts
- strong top-level hierarchy
- subtle motion only where it improves clarity
- rounded corners, but not excessively soft everywhere
- clear section separation and dependable component alignment

## Workflow

### 1. Understand the product and page purpose
Before changing the UI, determine:

- what the product does
- who the primary user is
- what action matters most on this screen
- what secondary actions exist
- what information is most important
- what the user should notice first, second, and third
- what business goal the screen supports

If the user did not provide every detail, infer sensible defaults and state assumptions briefly rather than stalling.

### 2. Audit the current UI or requested page
When improving an existing interface, identify:

- what feels unfinished
- what feels generic or AI-generated
- what is misaligned or visually inconsistent
- what hierarchy is weak
- what sections compete too much
- what interactions are unclear
- what content density is too high or too low
- which elements look decorative but not purposeful
- where the current experience breaks trust or flow

Separate findings into:

- visual issues
- UX issues
- content/structure issues
- implementation quality issues

### 3. Generate at least 3 real directions
Create at least 3 substantially different design approaches.

For each direction, define:

1. Core concept
2. Layout structure
3. Navigation model
4. Visual character
5. Key components used
6. Primary CTA strategy
7. Why a user would find it clear and appealing
8. Main trade-offs

Useful contrast patterns:

- Direction A: minimal and executive, reduced surface count
- Direction B: structured product dashboard, clearer grouping and utility
- Direction C: higher visual emphasis, stronger brand personality with controlled richness
- Direction D: workflow-first and action-oriented, optimized for speed

Do not let the directions collapse into near-duplicates.

### 4. Compare directions in prose
After exploring options, compare them on:

- clarity of hierarchy
- trustworthiness / product maturity
- distinctiveness
- implementation risk
- scalability across the product

Then choose one direction deliberately instead of blending everything together.

### 5. Implement with discipline
When implementing:

- keep the design system tight
- reuse strong patterns
- remove redundant text
- make primary actions obvious
- hide secondary detail behind disclosure when appropriate
- avoid overbuilding decorative UI

### 6. Final QA
Before finishing, check:

- alignment
- spacing rhythm
- button hierarchy
- text contrast
- responsive layout
- hover/focus/active states
- consistency across pages
- whether the result still feels too AI-generated
