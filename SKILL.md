# Startup UI/UX Design

Create interface work that feels like a real product made by a strong designer and frontend engineer, not a generic AI mockup.

This skill is inspired in part by the earlier interface-design skill’s strongest ideas: gather requirements first, generate multiple genuinely different options, compare them clearly, and synthesize the best final direction instead of locking into the first idea.

## Primary Goal

Produce UI/UX that is:

- visually polished
- structurally clear
- realistic for a startup product
- consistent across pages and states
- responsive and accessible
- implementable without breaking existing functionality

## Success Standard

A successful result should feel:

- intentional rather than autogenerated
- modern but not overdesigned
- clean, aligned, and trustworthy
- focused on the product’s main job
- coherent across layout, spacing, typography, color, states, and interactions

## Core Principles

### 1. Do not settle for the first idea
Generate at least 3 meaningfully different design directions before locking the final solution.

If sub-agents or parallel exploration tools are available, use them. If not, still reason through multiple alternatives manually.

Each direction must differ in a real way, not just by color or small layout changes. Differences may include:

- information hierarchy
- navigation model
- page density
- card vs table emphasis
- dashboard structure
- interaction pattern
- visual tone
- onboarding flow structure
- focus on analysis vs action

### 2. Preserve working product logic
When redesigning an existing interface:

- do not break working features just to restyle the UI
- keep important flows intact unless there is a strong UX reason to change them
- identify what is visual debt vs functional debt
- fix structure and usability before adding decoration

### 3. Prioritize clarity over decoration
Use visual styling to support understanding, not to distract.

Prefer:

- strong hierarchy
- good spacing
- readable typography
- clear primary actions
- obvious grouping
- consistent component behavior

Avoid:

- random gradients
- excessive glassmorphism
- too many accents competing for attention
- oversized shadows
- decorative elements with no UX purpose
- cluttered dashboards with equal emphasis everywhere

### 4. Make it feel like a real shipped product
Use realistic structure, labels, and content.

Do not produce pages full of vague placeholders such as:

- "Lorem ipsum"
- "Sample analytics"
- "Card title"
- "Insert data here"

Instead, use believable startup-oriented content and component labels that fit the product.

### 5. Design the states, not only the ideal screen
Every important flow should account for real UI states:

- default
- hover
- active
- focus
- selected
- disabled
- loading / skeleton
- empty
- success
- warning
- error
- no-results

A polished product is judged heavily by how these states are handled.

### 6. Responsive by design, not as an afterthought
Consider layout behavior from the start for:

- desktop
- tablet
- mobile

Specify what stacks, collapses, hides, scrolls, becomes tabbed, or moves into sheets/drawers.

### 7. Accessibility is part of good design
Ensure the design remains usable with:

- good color contrast
- visible focus states
- meaningful labels
- readable text sizes
- sensible hit targets
- keyboard-friendly interactions where relevant

### 8. Consistency beats novelty
A smaller number of well-used patterns is better than many one-off components.

Keep consistent:

- spacing scale
- border radius
- shadow depth
- input style
- button hierarchy
- icon sizing
- heading rhythm
- card behavior
- page paddings and gutters

## Default Design Taste

Unless the user explicitly asks otherwise, bias toward:

- one strong default theme instead of splitting effort across light and dark modes
- a clean modern SaaS aesthetic
- restrained but premium visuals
- medium density layouts
- strong top-level hierarchy
- subtle motion only where it improves clarity
- rounded corners, but not excessively soft everywhere
- clear section separation and dependable component alignment

## Workflow

### 1. Understand the product and page purpose
Before changing the UI, determine:

- what the product does
- who the primary user is
- what action matters most on this screen
- what secondary actions exist
- what information is most important
- what the user should notice first, second, and third
- what business goal the screen supports

If the user did not provide every detail, infer sensible defaults and state assumptions briefly rather than stalling.

### 2. Audit the current UI or requested page
When improving an existing interface, identify:

- what feels unfinished
- what feels generic or AI-generated
- what is misaligned or visually inconsistent
- what hierarchy is weak
- what sections compete too much
- what interactions are unclear
- what content density is too high or too low
- which elements look decorative but not purposeful
- where the current experience breaks trust or flow

Separate findings into:

- visual issues
- UX issues
- content/structure issues
- implementation quality issues

### 3. Generate at least 3 real directions
Create at least 3 substantially different design approaches.

For each direction, define:

1. Core concept
2. Layout structure
3. Navigation model
4. Visual character
5. Key components used
6. Primary CTA strategy
7. Why a user would find it clear and appealing
8. Main trade-offs

Useful contrast patterns:

- Direction A: minimal and executive, reduced surface count
- Direction B: structured product dashboard, clearer grouping and utility
- Direction C: higher visual emphasis, stronger brand personality with controlled richness
- Direction D: workflow-first and action-oriented, optimized for speed

Do not let the directions collapse into near-duplicates.

### 4. Compare directions in prose
After exploring options, compare them on:

- clarity of hierarchy
- trustworthiness / product maturity
- ease of scanning
- ease of action
- adaptability to future features
- implementation complexity
- risk of looking generic
- fit for the startup’s actual audience

Do not reduce this comparison to a shallow table only. Explain the reasoning.

### 5. Synthesize the final direction
Choose the best direction, then improve it by pulling in the strongest elements from the other options.

Examples:

- use Direction A’s cleaner header
- combine it with Direction B’s information grouping
- keep Direction C’s stronger hero or branding treatment
- retain Direction D’s faster action flow

The final design should not be an average compromise. It should be a deliberate best-of synthesis.

### 6. Translate design into implementation
When asked to implement:

- map the design to concrete components
- keep the code organized and production-minded
- avoid fragile one-off styling where reusable patterns make sense
- preserve working logic and routes
- make sure the final UI is complete, not halfway polished

Implementation should include:

- proper spacing and alignment
- consistent component variants
- responsive breakpoints
- empty/loading/error states where needed
- realistic labels and microcopy
- no broken pages or white screens after redesign

### 7. Self-review before finishing
Before presenting the final result, inspect for:

- alignment inconsistencies
- uneven spacing
- repeated card patterns that feel templated
- weak headers
- floating actions without context
- crowded sections
- too many accent colors
- contrast issues
- visual imbalance
- missing states
- anything that makes the design feel unfinished or fake

Revise before handing off.

## Design Standards by Surface

### Dashboard
A dashboard should answer, in order:

- what matters right now
- what changed
- what needs attention
- what the user can do next

Rules:

- do not give equal visual weight to every metric
- one or two primary insights should dominate
- filters must be understandable
- charts must serve decisions, not decoration
- support quick scanning before deeper exploration

### Landing Page
A landing page should quickly communicate:

- what the product is
- who it is for
- why it is valuable
- what action to take next

Rules:

- strong above-the-fold clarity
- believable product framing
- avoid inflated marketing fluff
- show product proof, workflow, or outcomes
- CTA should be obvious and well-placed

### Login / Auth
Rules:

- realistic product-grade form layout
- trustworthy spacing and alignment
- clear error handling
- useful secondary actions such as forgot password or sign up
- branding present but controlled
- no awkward empty surrounding space

### Onboarding
Rules:

- only ask what is needed
- make progress visible
- chunk decisions into manageable steps
- explain why information is requested when needed
- allow continuation without overwhelming the user

### Settings / Profile
Rules:

- group related controls logically
- separate account, preferences, privacy, billing, and integrations cleanly
- avoid long chaotic forms
- destructive actions must be clearly separated and confirmed

### Data Tables / Lists
Rules:

- optimize for scanability
- columns must be justified by real user need
- actions should be discoverable but not noisy
- support empty, filtered, and loading states elegantly

## Implementation Guidance for Frontend Code

When producing code, aim for:

- predictable layout structure
- reusable components
- minimal but effective motion
- accessible semantics
- no placeholder junk
- no inconsistent paddings between similar sections
- no accidental overflow or broken responsiveness
- no style drift between pages

If a design system is not already defined, establish lightweight defaults for:

- spacing scale
- typography scale
- radii
- shadows
- border colors
- background levels
- primary / secondary / destructive button treatments
- card patterns
- input patterns

## Output Format

When using this skill, prefer this response structure unless the user requests otherwise:

1. Product/page understanding
2. Key UX or visual problems
3. Three or more design directions
4. Comparison and recommendation
5. Final design system / layout decisions
6. Implementation plan or code changes
7. Final QA checklist

If the user explicitly wants direct implementation without a long concept phase, still do a brief internal multi-option check before coding.

## Anti-Patterns

- Do not just restyle one screen superficially.
- Do not make every section a card with identical weight.
- Do not use randomness as creativity.
- Do not produce design that looks good only in a screenshot but fails in real usage.
- Do not prioritize trends over clarity.
- Do not ask too many follow-up questions when reasonable defaults can be inferred.
- Do not leave broken functionality while polishing visuals.
- Do not produce pages that feel obviously AI-generated because of repetitive layout rhythm, generic copy, or unnatural spacing.

## Special Instruction for Existing Startup Projects

If the user is improving an already-built product:

- respect the current product direction
- identify what must be kept vs redesigned
- tighten the experience holistically rather than patching isolated components only
- fix the shell first when needed: header, navigation, page container, spacing system, action hierarchy
- then refine individual screens
- make the whole product feel like one coherent system

## Final Bar

Before concluding, ask yourself:

- Does this look like a product someone would trust?
- Does it feel designed, not autogenerated?
- Is the hierarchy obvious within 3 seconds?
- Are important actions impossible to miss?
- Is the layout balanced and aligned?
- Would this still feel good with real content and edge cases?

If any answer is no, iterate again before finalizing.
